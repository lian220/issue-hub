# 005: Airflow 도입 전략 (Phase 3 확정)

## Context

IssueHub는 이슈 통합, 양방향 동기화, 정책 관리, 자동화, SLA, 향후 AI/RAG 기능을 포함하는 플랫폼이다.

현재 아키텍처와 로드맵 기준의 핵심 백그라운드 작업은 아래와 같다.

- 웹훅 수신 후 비동기 후처리
- 커넥터 보완 폴링
- SLA 체크
- 정책 만료/갱신 알림
- 향후 정책 문서 임베딩 생성
- 감사 로그 아카이빙 가능성

이 과정에서 "Airflow를 지금부터 도입할 필요가 있는가?"와 "도입한다면 언제가 적절한가?"를 결정해야 한다.

## Decision

### 현재 Phase에서는 Airflow를 도입하지 않는다

Phase 1-2에서는 k3d CronJob으로 배치 작업을 처리하고, Phase 3에서 AI/RAG 다단계 파이프라인 도입과 함께 Airflow를 도입한다.
초기 배치/백그라운드 작업은 아래 방식으로 처리한다.

- `Spring Scheduler`
- `@Async`
- 작업 상태/실행 이력 테이블
- 필요 시 `Redis Streams`

### Airflow는 배치 오케스트레이션 수요가 커질 때만 검토한다

Airflow는 실시간 이벤트 처리나 동기화 엔진의 핵심 기술로 채택하지 않는다.
대신 아래와 같은 **다단계 배치 오케스트레이션** 수요가 실제로 커질 때 도입 후보로 검토한다.

- 초기 대량 적재 배치
- 정합성 검증 파이프라인
- 재색인 파이프라인
- 감사 로그 장기 보관 파이프라인
- 정책 문서 임베딩 재생성 배치
- AI 데이터셋 정제/평가 파이프라인
- 정기 리포트 생성 파이프라인

## Rationale

### 1. IssueHub의 초기 핵심은 실시간 처리다

현재 제품의 핵심 복잡도는 아래에 있다.

- 웹훅 기반 실시간 변경 감지
- 양방향 동기화
- 충돌 감지와 수동 해결
- 짧은 비동기 후처리

Airflow는 이벤트 브로커나 실시간 동기화 엔진이 아니라, 배치 워크플로우 오케스트레이터에 가깝다.
따라서 초기 핵심 문제를 푸는 데 직접적인 이득이 크지 않다.

### 2. 초기 팀 규모와 인프라 원칙에 비해 운영 부담이 크다

Airflow를 도입하면 아래 운영 요소가 추가된다.

- Scheduler
- Webserver
- Metadata DB
- Executor/Worker
- DAG 배포 및 버전 관리

현재 로드맵은 최소 인프라로 시작해 점진적으로 확장하는 전략이므로, Airflow는 초기 복잡도를 불필요하게 높일 가능성이 크다.

### 3. 도메인 로직이 앱 밖으로 분산될 위험이 있다

동기화, 정책, 알림 관련 핵심 로직 일부가 DAG로 이동하면 아래 문제가 생길 수 있다.

- 비즈니스 규칙이 Spring 애플리케이션과 Airflow에 이중 분산
- 테스트 경계가 불명확해짐
- 장애 추적 경로가 길어짐

초기에는 제품 핵심 로직을 애플리케이션 내부에 유지하는 편이 낫다.

## When Airflow Becomes Reasonable

아래 조건 중 여러 개가 실제로 충족되면 Airflow 도입을 검토한다.

### 1. 배치 파이프라인이 다단계 의존 구조로 커질 때

예:

`초기 동기화 -> 정합성 검증 -> 재색인 -> 리포트 생성 -> 알림 발송`

### 2. 실패 재처리와 운영 가시성이 핵심 요구가 될 때

단순 애플리케이션 로그나 cron 로그만으로는 운영이 어려운 수준이 되면, DAG 단위 상태 추적과 재실행 기능의 가치가 커진다.

### 3. AI/RAG 파이프라인이 독립 운영 단위가 될 때

예:

- 문서 수집
- 정제
- chunking
- embedding
- 인덱싱
- 평가
- 품질 리포트 생성

### 4. 장기 보관/감사성 배치가 정기적으로 돌아야 할 때

예:

- 감사 로그 아카이빙
- 규제 대응용 증적 리포트 생성
- 정책 준수율 정기 집계

## Phase Guidance

### Phase 1

도입하지 않는다.

- `Spring Scheduler + @Async`로 충분
- 동기화보다 제품 핵심 모델 안정화가 우선

### Phase 2

여전히 도입하지 않는다.

- 웹훅/폴링/양방향 동기화는 앱 내부에서 유지
- 필요 시 `Redis Streams`로 비동기 내구성만 보강

### Phase 3

Airflow를 도입한다. k3d 클러스터 위에 K8sExecutor로 배포한다.

- 정책 임베딩 재생성
- 감사 아카이빙
- 대량 문서 재처리
- AI 배치 파이프라인이 실제로 증가할 경우

도입 시 프로젝트 구조:

```
pipelines/          ← Python (Airflow DAG)
├── dags/
│   ├── embedding_refresh.py     # 전체 임베딩 재생성
│   ├── opensearch_reindex.py    # OpenSearch 전체 리빌드
│   ├── audit_archive.py         # 감사 로그 아카이빙
│   └── ai_data_pipeline.py     # AI 데이터 정제/평가
├── libs/
│   ├── embedding_client.py      # Gemini 임베딩 호출
│   └── issuehub_api_client.py   # IssueHub REST API 클라이언트
└── requirements.txt
```

### Phase 4+

배치가 독립 운영 계층으로 성장하면 도입 가치가 커진다.

- AI/RAG 운영 고도화
- 리포팅/아카이빙 확대
- 다수 외부 시스템 대상 정기 배치

## KEDA 관련 결정

전문가 패널 검토 결과, KEDA(Kubernetes Event-Driven Autoscaling)는 현 시점에서 도입하지 않는다.

### 보류 이유
- 50명 사내 플랫폼에서 이벤트 기반 0→1→0 스케일링이 해결하는 실제 문제가 부재
- K8s 기본 HPA(CPU/메모리 기반)로 오토스케일링 학습이 우선
- 의존성 체인이 가장 길어(K8s + Kafka + KEDA) 복잡도 대비 학습 효과가 낮음

### 도입 검토 시점
- Phase 4 이후, K8s와 Kafka가 안정화된 상태에서 선택적 심화 과제로 검토
- 또는 트래픽 변동이 실제로 커져 HPA만으로 부족한 경우

## Consequences

### Positive

- 초기 인프라 복잡도를 낮출 수 있다
- 핵심 도메인 로직을 애플리케이션 내부에 유지할 수 있다
- 실시간 동기화와 배치 오케스트레이션을 혼동하지 않게 된다

### Negative

- 배치 파이프라인이 커졌을 때 운영 가시성이 부족할 수 있다
- 재실행/이력 추적 기능을 애플리케이션에서 직접 설계해야 한다

## Follow-up

Airflow를 도입하지 않는 대신 아래를 먼저 준비한다.

1. 배치 작업 실행 이력 테이블 설계
2. 작업별 재시도/백오프 정책 표준화
3. 수동 재실행 API 또는 관리자 UI 제공
4. 배치 복잡도가 커질 경우 재평가 기준 문서화
