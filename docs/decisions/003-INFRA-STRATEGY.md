# ADR-003: 인프라 전략

## Context (배경)

IssueHub는 기존에 Jira, GitHub, Slack 등 분산된 도구를 사용하는 조직을 대상으로 하며, 도입 시 기존 시스템과의 병행 운영 및 점진적 마이그레이션이 필수적이다.

인프라 측면에서는 Phase별로 요구사항이 달라지므로, 초기에 과도한 인프라를 구축하지 않고 실제 필요에 따라 점진적으로 확장하는 전략이 필요하다. 이를 통해 운영 부담을 최소화하면서도 장기적으로는 대규모 트래픽과 복잡한 요구사항에 대응할 수 있는 구조를 갖추어야 한다.

## Decision (결정)

### 도입 및 마이그레이션 계획

#### 데이터 마이그레이션

- 기존 Jira/GitHub 데이터의 초기 동기화 배치 실행
- 마이그레이션 대상: 이슈, 댓글, 첨부파일, 라벨, 상태 이력
- 데이터 정합성 검증 절차 포함

#### 병행 운영

- Phase 1 배포 후 **4주간** 기존 도구와 IssueHub 병행 운영
- 병행 기간 중 데이터 일관성 모니터링 및 차이점 리포팅

#### 교육 계획

- 부서별 온보딩 세션 진행 (각 1~2시간)
- 사용자 가이드 및 FAQ 문서 제공
- 1:1 지원 채널 운영 (Slack 채널 또는 전담 담당자)

#### 점진적 도입 순서

1. **개발팀** (1~2주차): 이슈 통합, GitHub 연동 중심
2. **PM/기획** (3~4주차): 대시보드, 필터, 리포팅 중심
3. **CS/운영** (5~6주차): Slack 알림, 자동 라우팅 중심
4. **전사 확대** (7주차~): 정책 관리, 컴플라이언스 포함 전체 기능

#### Exit 전략

- 각 Phase 종료 시 **Go/No-Go 기준** 적용 (채택률, 만족도, 장애 빈도)
- 실패 시 데이터 역마이그레이션 절차 수행 (IssueHub → 기존 도구로 데이터 복원)
- Phase별 중단 비용 및 영향도 사전 산정

---

### 단계별 인프라 진화

초기에는 최소한의 인프라로 시작하여, 실제 필요에 따라 점진적으로 확장한다.

#### Phase 1: PostgreSQL + Redis (2개)

| 영역 | 기술 | 비고 |
|------|------|------|
| **이벤트** | Spring ApplicationEvent + @Async | 프로세스 내 비동기 처리 |
| **인증** | Spring Security + 자체 JWT | 별도 인프라 불필요 |
| **검색** | PostgreSQL LIKE + 기본 인덱스 | B-tree 인덱스 활용 |

#### Phase 2: PostgreSQL + Redis (2개, 기능 확장)

| 영역 | 기술 | 비고 |
|------|------|------|
| **이벤트** | Redis Streams (이벤트 영속성 필요 시) | 이벤트 재처리, 컨슈머 그룹 지원 |
| **인증** | 그대로 (Spring Security + JWT) | 변경 없음 |
| **검색** | PostgreSQL FTS (GIN 인덱스) | 한국어 형태소 분석 포함 |

#### Phase 3: PostgreSQL + Redis + Keycloak (3개)

| 영역 | 기술 | 비고 |
|------|------|------|
| **확장** | pgvector 확장 활성화 | RAG 기반 유사 이슈/정책 검색 |
| **인증** | Keycloak 도입 (SSO 필요 시) | OAuth2/OIDC, 소셜 로그인 |

#### Phase 4+: 필요 시 Kafka 도입

| 조건 | 기술 | 비고 |
|------|------|------|
| 일 수만 건 이상 이벤트 | Apache Kafka | 대규모 비동기 처리, 이벤트 소싱 |

## Consequences (결과)

### 긍정적 결과

- **운영 부담 최소화**: Phase 1에서 관리해야 할 인프라가 PostgreSQL과 Redis 2개뿐이므로 소규모 팀이 운영 가능
- **리스크 분산**: Phase별 Go/No-Go 기준을 통해 실패 시 빠르게 롤백할 수 있는 구조
- **조직 적응 시간 확보**: 부서별 점진적 도입으로 사용자 저항을 최소화하고 피드백 반영 가능
- **비용 예측 가능**: 각 Phase에서 필요한 인프라만 추가하므로 비용이 실제 사용량에 비례
- **데이터 안전성**: 역마이그레이션 절차를 사전에 준비하여 데이터 손실 리스크 방지

### 부정적 결과 및 제약

- **Phase 전환 비용**: 인프라 업그레이드 시 마이그레이션 작업이 필요하며, 일시적인 서비스 중단 가능성 존재
- **4주 병행 운영 부담**: 기존 도구와 IssueHub를 동시에 운영하는 기간 중 데이터 동기화 및 모니터링 리소스 필요
- **교육 비용**: 부서별 온보딩 세션 및 지원 채널 운영에 인력 투입 필요
- **Phase 4 불확실성**: Kafka 도입 시점과 조건이 명확하지 않아 장기 계획 수립에 한계
