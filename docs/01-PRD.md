# IssueHub - 제품 요구사항 정의서 (PRD)

> 문서 버전: 1.0
> 최종 수정일: 2026-03-21
> 작성자: IssueHub 개발팀
> 상태: 검토 중

---

## 1. 프로젝트 개요

**IssueHub**는 사내 이슈 통합 및 자동화 프로세스, 정책 관리를 위한 올인원 플랫폼이다.
Jira, GitHub, Slack, Teams 등 분산된 이슈 소스를 하나의 대시보드로 통합하고,
규칙 기반 자동화와 RAG 기반 AI 기능을 통해 이슈 관리 효율을 극대화한다.

---

## 2. 배경 및 문제 정의

### 2.1 현재 상황

| 문제 영역 | 상세 |
|-----------|------|
| **이슈 분산** | Jira, GitHub Issues, Slack 스레드, Teams 채널 등에 이슈가 흩어져 있어 전체 현황 파악 불가 |
| **수동 라우팅** | 이슈 접수 후 담당자 배정, 우선순위 판단, 에스컬레이션을 사람이 직접 수행 |
| **알림 누락** | 플랫폼별 알림 설정이 달라 중요 이슈 누락 빈번 |
| **정책 관리 부재** | 정책 문서가 Confluence, Google Docs, 로컬 파일 등에 산재, 버전 관리 및 컴플라이언스 추적 불가 |
| **SLA 추적 어려움** | SLA 위반 여부를 수동으로 확인, 에스컬레이션 지연 발생 |

### 2.2 해결 방향

하나의 플랫폼에서 모든 이슈를 통합 조회하고, 규칙 기반 자동화로 분류/라우팅/알림을 처리하며,
정책 문서의 전체 라이프사이클을 관리하고, RAG 기반 AI로 지능형 이슈 관리를 지원한다.

## 3. 대상 사용자

### 3.1 사용자 규모

- **조직 규모**: 10~50명
- **대상**: 전사 조직 (개발팀 + 기획/운영/CS 등 비기술 조직)

### 3.2 사용자 페르소나

| 페르소나 | 역할 | 주요 니즈 |
|----------|------|-----------|
| **개발 리드** | 기술 이슈 관리, 코드 리뷰 연동 | GitHub PR/Issue와 Jira 양방향 동기화, SLA 모니터링 |
| **PM/기획자** | 전체 이슈 현황 파악, 우선순위 조정 | 통합 대시보드, 트렌드 분석, 팀 워크로드 시각화 |
| **운영/CS** | 고객 이슈 접수 및 추적 | Slack/Teams 알림, 자동 분류, 에스컬레이션 |
| **컴플라이언스 담당** | 정책 관리 및 준수 추적 | 정책 버전 관리, 읽음 확인, 위반 감지 |
| **관리자** | 시스템 설정 및 권한 관리 | RBAC, 커넥터 설정, 자동화 규칙 관리 |

## 4. 핵심 목표

### 목표 1: 이슈 통합 허브

Jira와 GitHub의 이슈를 하나의 대시보드에서 통합 조회, 검색, 필터링한다.
각 플랫폼의 이슈를 정규화된 형태로 저장하고, 양방향 동기화를 지원한다.

### 목표 2: 자동화 엔진

규칙 기반 엔진을 통해 이슈 분류, 담당자 라우팅, 알림 발송, SLA 모니터링을 자동화한다.
트리거-조건-액션 구조로 유연한 자동화 규칙을 정의할 수 있다.

### 목표 3: 정책 관리

정책 문서의 CRUD, 버전 관리, 승인 워크플로우, 컴플라이언스 추적을 제공한다.
정책 위반 시 이슈를 자동 생성하고, 만료 알림을 발송한다.

### 목표 4: 외부 연동

Jira, GitHub, Slack, Teams와의 연동을 UI에서 설정하고 관리한다.
OAuth 기반 인증, 웹훅 수신, 필드 매핑을 지원한다.

### 목표 5: RAG 기반 AI 기능

정책 문서 기반 Q&A 봇, 유사 이슈 탐지, 이슈 자동 분류, 정책 위반 감지를 제공한다.
pgvector를 활용한 임베딩 기반 유사도 검색을 구현한다.

## 6. 비기능 요구사항

### 6.1 성능

| 항목 | 요구사항 |
|------|----------|
| API 응답 시간 | 95th percentile 500ms 이내 |
| 웹훅 처리 지연 | 5초 이내 |
| SSE 이벤트 전달 | 2초 이내 |
| 동시 사용자 | 50명 동시 접속 지원 |
| 이슈 검색 | 10만건 기준 1초 이내 |

### 6.2 실시간 업데이트

- Server-Sent Events (SSE) 기반 실시간 푸시
- Redis Pub/Sub를 통한 이벤트 분산
- 이슈 상태 변경, 댓글 추가, 동기화 완료 등 실시간 반영

### 6.3 배포 환경

- 온프레미스 배포 지원 (Docker Compose)
- 클라우드 배포 지원 (Kubernetes)
- 환경별 설정 분리 (dev, staging, production)

### 6.4 보안

| 항목 | 요구사항 |
|------|----------|
| 인증 | Keycloak 기반 OAuth2/OIDC |
| 토큰 저장 | AES-256 암호화 (외부 플랫폼 OAuth 토큰) |
| 웹훅 검증 | HMAC-SHA256 서명 검증 |
| API 인증 | JWT Bearer Token |
| 데이터 전송 | TLS 1.2 이상 |
| 감사 로그 | 모든 변경 작업 기록 |

### 6.5 가용성

- 목표 가용성: 99.5% (계획된 점검 제외)
- 장애 복구 시간(RTO): 4시간 이내
- 데이터 복구 시점(RPO): 1시간 이내

## 8. 제약사항 및 가정

### 8.1 제약사항

- Jira Cloud 및 GitHub Cloud 환경을 우선 지원 (Server/Data Center는 Phase 4 이후 검토)
- RAG 기반 AI 기능은 OpenAI API 또는 호환 LLM API 사용을 전제
- 단일 테넌트 구조로 시작 (멀티 테넌트는 향후 검토)

### 8.2 가정

- 대상 조직이 Jira, GitHub 중 하나 이상을 사용 중
- Keycloak 또는 호환 OIDC Provider 운영 가능
- Kafka 클러스터 운영 가능 (또는 Docker Compose로 로컬 실행)
- PostgreSQL 16 이상 사용 가능 (pgvector 확장 포함)

## 9. 용어 정의

| 용어 | 정의 |
|------|------|
| 이슈 | 외부 플랫폼(Jira, GitHub)의 작업 항목을 정규화한 내부 데이터 단위 |
