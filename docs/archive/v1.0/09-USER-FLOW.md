# IssueHub - 사용자 시점 워크플로우 (User Flow)

> 문서 버전: 1.0
> 최종 수정일: 2026-03-21
> 작성자: IssueHub AI 어시스턴트 (전문가 리뷰 기반 도출)
> 상태: 초안 (Draft)

---

## 1. 개요 (Overview)

이 문서는 기존 PRD, 시스템 아키텍처 및 로드맵 문서에는 구체적으로 명시되어 있지 않았던 **"사용자(User) 시점에서의 화면 흐름과 여정(User Journey)"**을 구체화한 문서입니다. 시스템 간의 데이터 흐름(Data Flow)이 아닌, **실제 조직 구성원들이 IssueHub를 화면에서 어떻게 클릭하고 인지하며 상호작용하는지**를 3가지 핵심 페르소나 케이스로 나누어 설명합니다.

---

## 2. 핵심 유저 페르소나 기반 시나리오

### Case 1: 운영/CS 담당자의 버그 접수 및 자동 라우팅 흐름 (Slack ➔ IssueHub ➔ Jira)

> **페르소나**: 커뮤니티 매니저(CS) 및 결제 백엔드 개발 리드
> **목적**: 흩어진 채널에서 들어온 VOC를 빠르게 개발팀에 전달하고 진행 상황을 투명하게 추적한다.

1. **이슈 접수 (Slack/Discord)**: 
   - CS 담당자가 커뮤니티 채널에서 "결제 진행 시 타임아웃 오류가 발생한다"는 유저 제보를 확인합니다.
   - 메신저 입력창에 `/issue create "결제 타임아웃 버그"`를 타이핑하거나, 메시지의 `[이슈로 만들기]` 인터랙티브 버튼을 클릭합니다.
2. **이슈 정제 및 자동화 개입 (IssueHub 엔진)**:
   - 메신저의 모달 창을 통해 상세 내용을 입력하고 `[제출]` 버튼을 누릅니다.
   - **(자동화 엔진 트리거)**: 내용을 분석한 IssueHub 엔진이 "결제" 키워드를 감지하여 우선순위를 `CRITICAL`로 변경하고, 대상 프로젝트를 결제 백엔드팀(Jira: PAY 프로젝트)으로 자동 지정합니다.
3. **개발 리드의 인지 및 배정 (대시보드)**:
   - 결제팀 개발 리드는 IssueHub의 **칸반 보드 뷰(Kanban Board View)**에서 `CRITICAL` 상태로 붉은색 알림이 깜빡이는 티켓을 발견합니다.
   - 드래그 앤 드롭으로 티켓을 특정 개발자(Assignee)에게 끌어다 놓습니다. 이 동작은 즉시 Jira 에 동기화(양방향 Sync)됩니다.
4. **이슈 해결 및 역방향 알림 (GitHub ➔ IssueHub ➔ Slack)**:
   - 담당 개발자가 코드를 수정하고 GitHub에 PR을 올리면서 `#PAY-102` (이슈 티켓 번호) 태그를 답니다.
   - 코드가 Merge되는 순간, GitHub 웹훅을 통해 IssueHub 티켓 상태가 `RESOLVED`로 자동 변경됩니다.
   - CS 담당자의 Slack 채널(또는 DM)에 즉시 "결제 타임아웃 버그가 해결되었습니다."라는 봇 알림 카드(Adaptive 카드의 형태)가 수신되어, 고객에게 빠르게 피드백을 전달합니다.

### Case 2: 보안 관리자의 정책 배포 및 컴플라이언스 준수 흐름

> **페르소나**: 보안/컴플라이언스 담당관 및 전사 임직원
> **목적**: 정책을 체계적으로 배포하고, 모든 구성원이 읽었는지 증적을 남기며, AI를 통해 정책 위반을 사전에 감지한다.

1. **정책 초안 생성 및 승인 (IssueHub 정책 탭)**:
   - 보안 담당관이 IssueHub 좌측 메뉴 `[정책 (Policies)]` 탭에 진입하여 `[새 정책 작성]` 버튼을 클릭합니다.
   - 마크다운 에디터에서 '26년도 상반기 개인정보 취급 방침 가이드'를 작성하고 상태를 `DRAFT`에서 `REVIEW`로 변경한 뒤, CISO(정보보호최고책임자)를 승인권자로 지정합니다.
   - CISO가 알림을 클릭해 내용을 확인 후 `[승인(APPROVE)]`ボタン을 누르면 최종적으로 `PUBLISHED(게시됨)` 상태가 됩니다.
2. **읽음 확인 프로세스 (Read-receipt)**:
   - 게재됨과 동시에 전사 직원의 Teams/Slack 채널에 정책 확인 요청 알림이 발송됩니다.
   - 직원은 IssueHub 정책 페이지 링크로 들어와 문서를 끝까지 스크롤한 뒤 `[읽음 확인]` 버튼을 명시적으로 클릭해야 합니다.
   - 보안 담당관은 대시보드의 **컴플라이언스 준수율 차트**에서 '개발팀 읽음 60% / 마케팅팀 90%' 식별하며 독려를 진행합니다.
3. **AI 기반 정책 위반 감지 (Phase 3 기능)**:
   - 이후 한 신입 개발자가 버그를 추적하기 위해 고객의 `주민등록번호`가 포함된 시스템 로그를 IssueHub 티켓 Description에 그대로 복사/붙여넣기 합니다.
   - 이슈 저장 버튼을 누르는 순간, 내부 AI(pgvector RAG)가 정책 위반 가능성을 감지하고 "⚠️ 입력하신 내용에 개인정보(주민등록번호)가 포함된 것으로 보입니다. 저장을 계속하시겠습니까?" 라는 **경고 팝업 모달**을 띄워 인적 오류(Human Error)를 방지합니다.

### Case 3: PM/기획자의 일간 프로젝트 진척도 모니터링 흐름

> **페르소나**: 프로젝트 매니저 (PM)
> **목적**: 분산된 모든 팀의 업무 진행 상황과 병목(Bottleneck) 구간을 한눈에 파악한다.

1. **모닝 브리핑 및 통합 대시보드 로드**:
   - PM이 데일리 스탠드업 미팅 전 IssueHub의 `[통합 대시보드]` 페이지에 접속합니다.
   - Jira(백엔드팀), GitHub(프론트엔드팀), 디자인 전용 툴에서 연동된 이슈가 **단일 타임라인 차트**로 노출됩니다.
2. **SLA 모니터링 및 병목 뷰어 확인**:
   - 위젯 중 **`SLA 위반(Breached)`** 항목에 집중합니다. 
   - 고객의 문의가 들어오거나 우선순위가 높은 작업이 24시간 넘게 `OPEN` 상태에 머물러 있는 리스트(전체 빨간색으로 하이라이팅)를 클릭하여 필터링합니다.
3. **팀 워크로드 분산 및 리-어사인 (Re-assign)**:
   - SLA 위반 티켓 5개가 모두 특정 A 개발자에게 몰려 있는 것을 하단 **팀 워크로드 차트**를 통해 시각적으로 발견합니다.
   - PM은 해당 개발자에게 지정된 티켓 박스들을 마우스로 드래그(Drag)하여 유휴 상태인 B 개발자의 Lane으로 이동시킵니다.
   - 이 변경 사항은 백그라운드 환경에서 각 플랫폼(GitHub/Jira)의 Assignee 변경 API를 호출해 자동으로 동기화됩니다.

## 3. UI/UX 디자인 진행을 위한 다음 단계 제안 (Next Steps)

현재 시스템 아키텍처와 엔드포인트(API) 스펙은 충분히 성숙해 있으나, 위 사용자 시나리오를 지원하기 위해서는 **명확한 화면 단위 와이어프레임(Wireframe)**이 필요합니다.

*   **필요 UI 화면 목록**:
    1. 통합 대시보드 현황판 (차트, SLA 알림패널 등)
    2. 칸반 (Kanban) 보드 뷰 (드래그 앤 드롭 지원)
    3. 정책 결재/열람 및 마크다운 에디터 화면
    4. 외부 커넥터 연동/설정 모달 (Settings)
    5. 슬랙/디스코드 등 메신저에서의 Adaptive Card UI (앱 내 UI는 아니지만 디자인 리소스 필요)
