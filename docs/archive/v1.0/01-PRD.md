# IssueHub - 제품 요구사항 정의서 (PRD)

> 문서 버전: 1.2
> 최종 수정일: 2026-03-21
> 작성자: IssueHub 개발팀
> 상태: 검토 중

---

## 1. 프로젝트 개요

**IssueHub**는 사내 이슈 통합 및 자동화 프로세스, 정책 관리를 위한 올인원 플랫폼이다.
Jira, GitHub, Slack, Teams, Discord 등 분산된 이슈 소스를 하나의 대시보드로 통합하고,
규칙 기반 자동화와 RAG 기반 AI 기능을 통해 이슈 관리 효율을 극대화한다.

---

## 2. 배경 및 문제 정의

### 2.1 현재 상황


| 문제 영역          | 상세                                                                 |
| -------------- | ------------------------------------------------------------------ |
| **이슈 분산**      | Jira, GitHub Issues, Slack 스레드, Teams 채널, Discord 채널 등에 이슈가 흩어져 있어 전체 현황 파악 불가 |
| **수동 라우팅**     | 이슈 접수 후 담당자 배정, 우선순위 판단, 에스컬레이션을 사람이 직접 수행                         |
| **알림 누락**      | 플랫폼별 알림 설정이 달라 중요 이슈 누락 빈번                                         |
| **정책 관리 부재**   | 정책 문서가 Confluence, Google Docs, 로컬 파일 등에 산재, 버전 관리 및 컴플라이언스 추적 불가  |
| **SLA 추적 어려움** | SLA 위반 여부를 수동으로 확인, 에스컬레이션 지연 발생                                   |


### 2.2 해결 방향

하나의 플랫폼에서 모든 이슈를 통합 조회하고, 규칙 기반 자동화로 분류/라우팅/알림을 처리하며,
정책 문서의 전체 라이프사이클을 관리하고, RAG 기반 AI로 지능형 이슈 관리를 지원한다.

## 4. 핵심 목표

### 목표 1: 이슈 통합 허브

Jira와 GitHub의 이슈를 하나의 대시보드에서 통합 조회, 검색, 필터링한다.
각 플랫폼의 이슈를 정규화된 형태로 저장하고, 양방향 동기화를 지원한다.

### 목표 2: 자동화 엔진

규칙 기반 엔진을 통해 이슈 분류, 담당자 라우팅, 알림 발송, SLA 모니터링을 자동화한다.
트리거-조건-액션 구조로 유연한 자동화 규칙을 정의할 수 있다.

### 목표 3: 정책 관리

정책 문서의 CRUD, 버전 관리, 승인 워크플로우, 컴플라이언스 추적을 제공한다.
정책 위반 시 이슈를 자동 생성하고, 만료 알림을 발송한다.

### 목표 4: 외부 연동

Jira, GitHub, Slack, Teams, Discord와의 연동을 UI에서 설정하고 관리한다.
OAuth 기반 인증, 웹훅 수신, 필드 매핑을 지원한다.

### 목표 5: RAG 기반 AI 기능

정책 문서 기반 Q&A 봇, 유사 이슈 탐지, 이슈 자동 분류, 정책 위반 감지를 제공한다.
pgvector를 활용한 임베딩 기반 유사도 검색을 구현한다.

#### 5.2.1 정책 승인 워크플로우 상세

> 전문가 패널 리뷰 후 확정된 상세 정책

| 항목 | 정책 |
| --- | --- |
| **승인자 지정** | 작성자가 직접 지정 + 역할 기반 자동 추천 (Manager 이상) 병행 |
| **승인자 수** | 다수 승인 (과반수 이상 승인 시 통과) |
| **반려 흐름** | 반려 시 반려 사유 필수 작성 → 정책 상태 DRAFT로 복귀 → 작성자에게 알림 |
| **게시 후 수정** | 게시된 정책 수정 가능, 단 재승인 필요 (새 버전 생성 + 승인 워크플로우 재시작) |
| **긴급 게시** | Admin 역할은 승인 워크플로우를 건너뛰고 즉시 게시 가능 (감사 로그 필수 기록) |

**승인 상태 흐름:**
```
DRAFT → REVIEW (검토 요청) → APPROVED (과반수 승인) → PUBLISHED (게시)
  ↑                              │
  └──── REJECTED (반려 사유 포함) ◄──┘
```

#### 5.3.1 양방향 동기화 충돌 해결 정책

> 전문가 패널 리뷰 후 확정된 동기화 전략

| 항목 | 정책 |
| --- | --- |
| **기본 전략** | 필드 단위 머지 (Field-Level Merge) |
| **서로 다른 필드 변경** | 자동 머지 (예: IssueHub에서 제목 수정 + Jira에서 담당자 변경 → 둘 다 반영) |
| **같은 필드 충돌** | CONFLICT 플래그 → 사용자에게 알림 → 수동 해결 (어느 쪽 선택할지 UI 제공) |
| **충돌 감지** | `sync_version` 기반 낙관적 잠금 |
| **충돌 해결 UI** | 양쪽 값 비교 화면 제공, 사용자가 선택 또는 직접 수정 |
| **미해결 충돌** | 충돌 목록 대시보드에서 관리, 미해결 건 알림 발송 |

## 10. 기술 스택 재조정 (전문가 권고 반영)

| **Keycloak** | Phase 1부터 인증 서버로 사용         | Phase 1: Spring Security + 자체 JWT 인증. Keycloak은 SSO 요구가 있는 Phase 3에서 도입                            |
| **pgvector** | Phase 1부터 PostgreSQL 확장 활성화 | Phase 1에서 제외. Phase 3 RAG 기능 도입 시에만 활성화                                                            |
| **검색 엔진**   | PostgreSQL FTS              | OpenSearch (로컬 Docker) 사용. nori 플러그인으로 한국어 형태소 분석 지원                                                |

### 10.1 인프라 구성 (확정)

> 전문가 패널 리뷰 후 확정된 인프라 전략

| 서비스 | 위치 | 제공자 | 사유 |
| --- | --- | --- | --- |
| **PostgreSQL** | 원격 (클라우드) | Aiven 무료 플랜 (5 GB) | 원본 데이터 보존, 로컬 환경 충돌 방지 |
| **Redis** | 원격 (클라우드) | Upstash 무료 플랜 (256 MB) | 캐시/세션, 유실 방지 |
| **OpenSearch** | 로컬 Docker | 자체 운영 | 검색 인덱스용, 유실 시 PostgreSQL에서 재빌드 가능 |

### 10.2 백업 전략

| 항목 | 방법 |
| --- | --- |
| **PostgreSQL** | `pg_dump` + GitHub Actions 자동화 (일 1회) |
| **Redis** | Upstash 자체 관리 (캐시 데이터이므로 유실 허용) |
| **OpenSearch** | 백업 불필요 (PostgreSQL에서 인덱스 재생성 가능) |

**결과**: 원격 인프라는 **Aiven PostgreSQL + Upstash Redis**, 로컬 인프라는 **OpenSearch (Docker)** 로 구성한다.

## 11. Build vs Buy 분석

> 자체 개발 의사결정의 근거를 명확히 하기 위한 비교 분석

### 11.1 Buy 옵션


| 구성                                                                   | 비용 (월)   | 커버리지       |
| -------------------------------------------------------------------- | -------- | ---------- |
| Jira Service Management + Jira Automation + Confluence + Zapier/Make | $200~500 | 기능의 70~80% |


### 11.2 Build이 정당한 경우

1. **조직 고유 워크플로우**: 정책-이슈 연계 워크플로우가 기존 SaaS로 구현 불가능한 경우
2. **보안/컴플라이언스**: 데이터 외부 전송이 불가능한 규제 환경 (온프레미스 필수)
3. **비용 효율성**: 50명 이상 확장 시 SaaS 라이선스 비용 급증이 예상되는 경우

### 11.3 하이브리드 접근 (권장)

- **기존 도구 활용**: 이슈 통합/자동화는 Jira, GitHub 등 기존 도구의 API를 최대한 활용
- **자체 개발 집중**: 정책 관리 + 컴플라이언스 추적 등 기존 도구로 커버되지 않는 영역만 자체 개발
- **점진적 전환**: Phase별 Go/No-Go 판단을 통해 Build 범위를 확대 또는 축소

## 12. 도입 및 마이그레이션 계획

### 12.1 데이터 마이그레이션

- 기존 Jira/GitHub 데이터의 초기 동기화 배치 실행
- 마이그레이션 대상: 이슈, 댓글, 첨부파일, 라벨, 상태 이력
- 데이터 정합성 검증 절차 포함

### 12.2 병행 운영

- Phase 1 배포 후 **4주간** 기존 도구와 IssueHub 병행 운영
- 병행 기간 중 데이터 일관성 모니터링 및 차이점 리포팅

### 12.3 교육 계획

- 부서별 온보딩 세션 진행 (각 1~2시간)
- 사용자 가이드 및 FAQ 문서 제공
- 1:1 지원 채널 운영 (Slack 채널 또는 전담 담당자)

### 12.4 점진적 도입 순서

1. **개발팀** (1~2주차): 이슈 통합, GitHub 연동 중심
2. **PM/기획** (3~4주차): 대시보드, 필터, 리포팅 중심
3. **CS/운영** (5~6주차): Slack 알림, 자동 라우팅 중심
4. **전사 확대** (7주차~): 정책 관리, 컴플라이언스 포함 전체 기능

### 12.5 Exit 전략

- 각 Phase 종료 시 **Go/No-Go 기준** 적용 (채택률, 만족도, 장애 빈도)
- 실패 시 데이터 역마이그레이션 절차 수행 (IssueHub → 기존 도구로 데이터 복원)
- Phase별 중단 비용 및 영향도 사전 산정
