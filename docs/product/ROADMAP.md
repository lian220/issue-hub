## 3. Definition of Done

모든 스프린트의 태스크는 아래 기준을 충족해야 완료로 인정한다.

```
[ ] 코드가 main에 머지됨
[ ] 단위 테스트 커버리지 80%+ (core 모듈)
[ ] API 통합 테스트 존재
[ ] OpenAPI 명세 최신화
[ ] 코드 리뷰 완료 (1명 이상)
[ ] API 응답 500ms 이내
[ ] 배포 파이프라인 통과
```

## 8. 리스크 관리

### 8.1 기술 리스크

| 리스크 | 발생 확률 | 영향도 | 대응 방안 |
|--------|-----------|--------|-----------|
| Jira/GitHub API Rate Limit 초과 | 중 | 높음 | 요청 큐잉, 지수 백오프, 벌크 API 활용 |
| pgvector 대규모 데이터 성능 | 중 | 중 | IVFFlat 리스트 수 튜닝, 배치 인덱싱 |
| 웹훅 수신 순서 보장 불가 | 중 | 중 | 이벤트 타임스탬프 기반 정렬, 멱등성 보장 |
| OAuth 토큰 만료 갱신 실패 | 중 | 높음 | 자동 갱신 로직, 실패 시 관리자 알림 |
| 외부 API 사양 변경 | 낮 | 중 | 커넥터 어댑터 패턴으로 격리, API 버전 고정 |

### 8.2 일정 리스크

| 리스크 | 발생 확률 | 영향도 | 대응 방안 |
|--------|-----------|--------|-----------|
| 기능 범위 확대 (Scope Creep) | 높 | 높음 | PRD 기준 엄격한 범위 관리, Phase 단위 리뷰 |
| 핵심 인력 이탈 | 낮 | 높음 | 페어 프로그래밍, 문서화, 코드 리뷰 강화 |
| 인프라 구축 지연 | 중 | 중 | Docker Compose 로컬 환경 우선, 클라우드 병행 준비 |

### 8.3 비즈니스 리스크

| 리스크 | 발생 확률 | 영향도 | 대응 방안 |
|--------|-----------|--------|-----------|
| 사용자 채택률 부진 | 중 | 높음 | Phase 1부터 개발팀 사용, 피드백 반영, 점진적 확대 |
| 기존 도구 대비 가치 부족 | 중 | 높음 | 통합 뷰와 자동화의 차별 가치 지속 홍보 |
| 보안 사고 | 낮 | 높음 | AES-256 암호화, HMAC 검증, 감사 로그, 정기 보안 점검 |

### 8.4 Phase별 Go/No-Go 기준

| Phase | Go/No-Go 기준 | 판단 시점 |
|-------|---------------|-----------|
| Phase 1 | 개발팀 5명 이상이 일주일간 실제 사용 후 긍정 피드백 | W7 |
| Phase 2 | 전사 시범 운영 2주 후 채택률 70% 이상 | W17 |
| Phase 3 | 전사 정식 운영 전환 승인 | W27 |
| Phase 4 | 500 동시 사용자 부하 테스트 통과, 1주 무장애 운영 | W33 |

### 8.5 Exit 전략

- **실패 시 데이터 역마이그레이션 절차**: 각 Phase의 Flyway 마이그레이션에 대응하는 롤백 스크립트를 사전 작성하여, 중단 시 이전 Phase 상태로 데이터를 복원할 수 있도록 한다.
- **외부 API 변경 리스크 대응**: 모든 외부 연동은 커넥터 어댑터 패턴(ConnectorAdapter 인터페이스)으로 격리하여, API 변경 시 어댑터 구현체만 교체하면 내부 로직에 영향이 없도록 설계한다.
