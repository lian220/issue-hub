# IssueHub - 제품 요구사항 정의서 (PRD)

> 문서 버전: 3.0
> 최종 수정일: 2026-03-23
> 작성자: IssueHub 개발팀
> 상태: 검토 중

---

## 1. 프로젝트 개요

**IssueHub**는 사내 이슈 통합 및 자동화 프로세스, 정책 관리를 위한 올인원 플랫폼이다.
Jira, GitHub, Slack, Teams, Discord 등 분산된 이슈 소스를 하나의 대시보드로 통합하고,
규칙 기반 자동화와 RAG 기반 AI 기능을 통해 이슈 관리 효율을 극대화한다.

v3.0부터 **Git 레포 코드 분석 + AI 기반 티켓 품질 관리** 기능을 추가하여,
코드베이스와 이슈를 연결한 지능형 분석을 제공한다.
**코딩 에이전트 연동** (OpenHands, SWE-agent 등)을 통해 티켓 기반 자동 개발 및 PR 생성을 지원하며,
**3가지 배포 모드** (셀프호스팅 / 하이브리드 에이전트 / 완전 SaaS)로 다양한 환경에 유연하게 대응한다.

## 2. 배경 및 문제 정의

### 2.1 현재 상황


| 문제 영역          | 상세                                                                 |
| -------------- | ------------------------------------------------------------------ |
| **이슈 분산**      | Jira, GitHub Issues, Slack 스레드, Teams 채널, Discord 채널 등에 이슈가 흩어져 있어 전체 현황 파악 불가 |
| **수동 라우팅**     | 이슈 접수 후 담당자 배정, 우선순위 판단, 에스컬레이션을 사람이 직접 수행                         |
| **알림 누락**      | 플랫폼별 알림 설정이 달라 중요 이슈 누락 빈번                                         |
| **정책 관리 부재**   | 정책 문서가 Confluence, Google Docs, 로컬 파일 등에 산재, 버전 관리 및 컴플라이언스 추적 불가  |
| **SLA 추적 어려움** | SLA 위반 여부를 수동으로 확인, 에스컬레이션 지연 발생                                   |


### 2.2 해결 방향

하나의 플랫폼에서 모든 이슈를 통합 조회하고, 규칙 기반 자동화로 분류/라우팅/알림을 처리하며,
정책 문서의 전체 라이프사이클을 관리하고, RAG 기반 AI로 지능형 이슈 관리를 지원한다.

## 4. 핵심 목표

### 목표 1: 이슈 통합 허브

Jira와 GitHub의 이슈를 하나의 대시보드에서 통합 조회, 검색, 필터링한다.
각 플랫폼의 이슈를 정규화된 형태로 저장하고, 양방향 동기화를 지원한다.

### 목표 2: 자동화 엔진

규칙 기반 엔진을 통해 이슈 분류, 담당자 라우팅, 알림 발송, SLA 모니터링을 자동화한다.
트리거-조건-액션 구조로 유연한 자동화 규칙을 정의할 수 있다.

### 목표 3: 정책 관리

정책 문서의 CRUD, 버전 관리, 승인 워크플로우, 컴플라이언스 추적을 제공한다.
정책 위반 시 이슈를 자동 생성하고, 만료 알림을 발송한다.

### 목표 4: 외부 연동

Jira, GitHub, Slack, Teams, Discord와의 연동을 UI에서 설정하고 관리한다.
OAuth 기반 인증, 웹훅 수신, 필드 매핑을 지원한다.

### 목표 5: RAG 기반 AI 기능

정책 문서 기반 Q&A 봇, 유사 이슈 탐지, 이슈 자동 분류, 정책 위반 감지를 제공한다.
pgvector를 활용한 임베딩 기반 유사도 검색을 구현한다.

#### 5.2.1 정책 승인 워크플로우 상세

> 전문가 패널 리뷰 후 확정된 상세 정책

| 항목 | 정책 |
| --- | --- |
| **승인자 지정** | 작성자가 직접 지정 + 역할 기반 자동 추천 (Manager 이상) 병행 |
| **승인자 수** | 다수 승인 (과반수 이상 승인 시 통과) |
| **반려 흐름** | 반려 시 반려 사유 필수 작성 → 정책 상태 DRAFT로 복귀 → 작성자에게 알림 |
| **게시 후 수정** | 게시된 정책 수정 가능, 단 재승인 필요 (새 버전 생성 + 승인 워크플로우 재시작) |
| **긴급 게시** | Admin 역할은 승인 워크플로우를 건너뛰고 즉시 게시 가능 (감사 로그 필수 기록) |

**승인 상태 흐름:**
```
DRAFT → REVIEW (검토 요청) → APPROVED (과반수 승인) → PUBLISHED (게시)
  ↑                              │
  └──── REJECTED (반려 사유 포함) ◄──┘
```

**추가 승인 규칙 (미확정 → 구현 시 확정 필요):**

| 항목 | 후보 정책 | 상태 |
| --- | --- | --- |
| 자가 승인 금지 | 정책 작성자는 자기 정책의 승인자가 될 수 없음 | 미확정 |
| 승인 범위 | Manager는 자기 팀 관련 정책만 승인 가능. 전사 정책은 Admin만 승인 | 미확정 |
| 긴급 게시 사후 검토 | Admin 긴급 게시 후 7일 이내 사후 승인 필수. 미승인 시 자동 비공개 전환 | 미확정 |

#### 5.3.1 양방향 동기화 충돌 해결 정책

> 전문가 패널 리뷰 후 확정된 동기화 전략

| 항목 | 정책 |
| --- | --- |
| **기본 전략** | 필드 단위 머지 (Field-Level Merge) |
| **서로 다른 필드 변경** | 자동 머지 (예: IssueHub에서 제목 수정 + Jira에서 담당자 변경 → 둘 다 반영) |
| **같은 필드 충돌** | CONFLICT 플래그 → 사용자에게 알림 → 수동 해결 (어느 쪽 선택할지 UI 제공) |
| **충돌 감지** | `sync_version` 기반 낙관적 잠금 |
| **충돌 해결 UI** | 양쪽 값 비교 화면 제공, 사용자가 선택 또는 직접 수정 |
| **미해결 충돌** | 충돌 목록 대시보드에서 관리, 미해결 건 알림 발송 |

**추가 동기화 규칙 (미확정 → 구현 시 확정 필요):**

| 항목 | 후보 정책 | 상태 |
| --- | --- | --- |
| 외부 이슈 삭제 시 | IssueHub에서 soft delete 처리 + "외부에서 삭제됨" 태그 부착. 동기화 중단 | 미확정 |
| 웹훅 멱등성 | 외부 이벤트 ID + 타임스탬프 기반 중복 방지. webhook_event 테이블에 수신 이력 저장 | 미확정 |
| 이벤트 순서 역전 | 이벤트 타임스탬프 기반 정렬. 오래된 이벤트가 늦게 도착 시 무시 (sync_version 비교) | 미확정 |
| 댓글/첨부파일 충돌 | 댓글은 append-only (충돌 없음). 첨부파일은 외부 링크만 저장 (원본 미보관) | 미확정 |
| Source of Truth | 외부 플랫폼이 원본(source of truth). IssueHub는 정규화된 사본 + 부가 기능 계층 | 미확정 |

## 13. 용어 정의


| 용어     | 정의                                          |
| ------ | ------------------------------------------- |
| 이슈     | 외부 플랫폼(Jira, GitHub)의 작업 항목을 정규화한 내부 데이터 단위 |

| 자동화 규칙 | 트리거-조건-액션으로 구성된 이벤트 기반 자동 처리 규칙             |
| SLA    | Service Level Agreement, 이슈 응답/해결 목표 시간     |
| RAG    | Retrieval-Augmented Generation, 검색 증강 생성    |
| 동기화 버전 | 외부 플랫폼과의 데이터 일관성을 추적하기 위한 버전 번호             |
