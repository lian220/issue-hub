#### UC-ISS-001: 통합 이슈 목록 조회

| 항목 | 내용 |
|------|------|
| **UC ID** | UC-ISS-001 |
| **이름** | 통합 이슈 목록 조회 (필터, 검색, 정렬) |
| **액터** | Member, Manager, Admin, Viewer |
| **사전 조건** | 사용자가 로그인 상태이며, 1개 이상의 커넥터가 연결되어 있다. |

**기본 플로우:**

1. 사용자가 좌측 메뉴에서 `[이슈]`를 클릭한다.
2. 시스템이 사용자의 접근 가능한 프로젝트의 전체 이슈 목록을 로드한다.
3. 이슈 목록이 기본 정렬(최신순)로 표시된다. 각 이슈에 소스 플랫폼 아이콘(Jira/GitHub), 상태 뱃지, 우선순위, 담당자가 표시된다.
4. 사용자가 필터 패널에서 조건을 선택한다 (소스 플랫폼, 상태, 우선순위, 담당자, 날짜 범위, 라벨).
5. 시스템이 필터 조건에 맞는 이슈 목록을 재조회하여 표시한다.
6. 사용자가 정렬 기준을 변경한다 (생성일, 수정일, 우선순위, SLA 마감일).
7. 시스템이 정렬 기준에 따라 목록을 재정렬한다.

**대안 플로우:**

- **3a.** 조회 결과가 0건인 경우, "조건에 맞는 이슈가 없습니다" 안내와 함께 필터 초기화 버튼을 표시한다.
- **4a.** 사용자가 저장된 필터 프리셋을 선택하면, 해당 조건이 자동 적용된다.
- **5a.** 페이지네이션이 적용되어 한 번에 최대 50건을 표시하며, 스크롤 시 추가 로드한다.

**사후 조건:** 사용자가 필터/정렬 조건에 맞는 이슈 목록을 화면에서 확인한다.

**관련 화면:** `/issues`

#### UC-ISS-002: 이슈 상세 조회

| 항목 | 내용 |
|------|------|
| **UC ID** | UC-ISS-002 |
| **이름** | 이슈 상세 조회 (댓글, 변경 이력, 연결된 PR) |
| **액터** | Member, Manager, Admin, Viewer |
| **사전 조건** | 이슈 목록에서 특정 이슈를 식별한 상태이다. |

**기본 플로우:**

1. 사용자가 이슈 목록에서 특정 이슈를 클릭한다.
2. 시스템이 이슈 상세 페이지를 로드한다. 표시 항목: 제목, 설명, 상태, 우선순위, 담당자, 라벨, 생성일, 수정일, 소스 플랫폼 링크.
3. 하단 탭에서 `[댓글]` 탭이 기본 선택되어 댓글 목록이 표시된다. 외부 플랫폼의 동기화된 댓글에는 소스 아이콘이 표시된다.
4. 사용자가 `[변경 이력]` 탭을 클릭하면, 필드별 변경 내역(이전 값, 새 값, 변경자, 시간)이 타임라인 형태로 표시된다.
5. 사용자가 `[연결된 PR]` 탭을 클릭하면, GitHub PR 목록(제목, 상태, 리뷰어, 머지 여부)이 표시된다.
6. 우측 사이드바에 이슈 메타데이터(관계 이슈, 워치 수, SLA 정보)가 표시된다.

**대안 플로우:**

- **5a.** 연결된 PR이 없는 경우, "연결된 Pull Request가 없습니다" 메시지를 표시한다.
- **6a.** SLA가 설정된 이슈인 경우, 남은 시간이 카운트다운으로 표시된다. 위반 시 빨간색으로 강조된다.

**사후 조건:** 사용자가 이슈의 모든 상세 정보를 확인한다.

**관련 화면:** `/issues/{id}`

#### UC-ISS-003: 이슈 생성

| 항목 | 내용 |
|------|------|
| **UC ID** | UC-ISS-003 |
| **이름** | 이슈 생성 (템플릿 선택, 폼 작성, 담당자 배정) |
| **액터** | Member, Manager, Admin |
| **사전 조건** | 사용자가 이슈 생성 권한(Member 이상)을 보유하고 있다. |

**기본 플로우:**

1. 사용자가 `[새 이슈]` 버튼을 클릭한다.
2. 시스템이 템플릿 선택 화면을 표시한다 (버그 리포트, 기능 요청, 고객 문의, 빈 양식).
3. 사용자가 템플릿을 선택한다.
4. 시스템이 선택된 템플릿의 폼을 표시한다. 필수 항목: 제목, 설명. 선택 항목: 우선순위, 라벨, 담당자, 대상 프로젝트.
5. 사용자가 폼을 작성하고 담당자를 배정한다.
6. 사용자가 `[생성]` 버튼을 클릭한다.
7. 시스템이 유사 이슈 탐지(UC-ISS-008)를 수행하여 결과가 있으면 알림을 표시한다.
8. 사용자가 유사 이슈 확인 후 생성을 확정한다.
9. 시스템이 이슈를 DB에 저장하고, 연결된 커넥터로 아웃바운드 동기화를 수행한다.
10. 이벤트가 발행되어 자동화 규칙 평가 및 알림 발송이 트리거된다.
11. 이슈 상세 페이지로 이동한다.

**대안 플로우:**

- **7a.** 유사 이슈가 탐지되지 않으면, 바로 생성을 진행한다.
- **7b.** 사용자가 유사 이슈 중 하나를 선택하면, 해당 이슈 상세 페이지로 이동한다 (중복 방지).
- **9a.** 아웃바운드 동기화에 실패하면, 이슈는 로컬에 저장되고 `sync_status=PENDING`으로 마킹한다. 재시도 큐에 등록한다.

**사후 조건:** 새 이슈가 생성되고 연동 플랫폼에 동기화된다. 담당자에게 알림이 발송된다.

**관련 화면:** `/issues/new`

#### UC-ISS-004: 이슈 상태 변경

| 항목 | 내용 |
|------|------|
| **UC ID** | UC-ISS-004 |
| **이름** | 이슈 상태 변경 (OPEN -> IN_PROGRESS -> RESOLVED -> CLOSED) |
| **액터** | Member, Manager, Admin |
| **사전 조건** | 대상 이슈가 존재하며, 사용자에게 수정 권한이 있다. |

**기본 플로우:**

1. 사용자가 이슈 상세 페이지에서 현재 상태 뱃지를 클릭한다.
2. 시스템이 전환 가능한 상태 목록을 드롭다운으로 표시한다.
3. 사용자가 새 상태를 선택한다.
4. 시스템이 상태 전환 유효성을 검증한다 (허용된 전환인지 확인).
5. 상태가 변경되고, 변경 이력이 기록된다.
6. 이벤트가 발행되어 아웃바운드 동기화, 자동화 규칙 평가, 알림 발송이 트리거된다.
7. SLA가 설정된 이슈인 경우, SLA 시계가 상태에 따라 시작/정지/종료된다.

**대안 플로우:**

- **4a.** 유효하지 않은 상태 전환(예: CLOSED -> IN_PROGRESS)인 경우, "이 상태로 전환할 수 없습니다" 메시지를 표시한다.
- **6a.** 동기화 충돌이 발생하면, 충돌 해결 플로우(2.1.4)로 진입한다.

**사후 조건:** 이슈 상태가 변경되고 연동 플랫폼에 반영된다.

**관련 화면:** `/issues/{id}`

#### UC-ISS-005: 칸반 보드에서 드래그앤드롭 상태 변경

| 항목 | 내용 |
|------|------|
| **UC ID** | UC-ISS-005 |
| **이름** | 칸반 보드에서 드래그앤드롭 상태 변경 |
| **액터** | Member, Manager, Admin |
| **사전 조건** | 사용자가 칸반 보드 뷰에 접근한 상태이다. |

**기본 플로우:**

1. 사용자가 이슈 목록 페이지에서 `[칸반 보드]` 뷰를 선택한다.
2. 시스템이 상태별 컬럼(OPEN, IN_PROGRESS, RESOLVED, CLOSED)으로 이슈 카드를 표시한다.
3. 사용자가 이슈 카드를 드래그하여 다른 상태 컬럼에 드롭한다.
4. 시스템이 상태 전환 유효성을 검증한다.
5. 상태가 변경되고, 카드가 새 컬럼에 고정된다.
6. 아웃바운드 동기화 및 알림이 트리거된다.
7. SSE를 통해 같은 보드를 보고 있는 다른 사용자에게 실시간으로 변경이 반영된다.

**대안 플로우:**

- **4a.** 유효하지 않은 상태 전환인 경우, 카드가 원래 위치로 되돌아가고 에러 토스트가 표시된다.
- **3a.** 사용자가 같은 컬럼 내에서 카드를 이동하면, 표시 순서만 변경된다.

**사후 조건:** 이슈 상태가 변경되고, 보드 UI가 업데이트된다.

**관련 화면:** `/issues/board`

#### UC-ISS-006: 이슈 검색 (전문 검색)

| 항목 | 내용 |
|------|------|
| **UC ID** | UC-ISS-006 |
| **이름** | 이슈 검색 (전문 검색) |
| **액터** | Member, Manager, Admin, Viewer |
| **사전 조건** | 이슈가 1건 이상 존재한다. |

**기본 플로우:**

1. 사용자가 상단 검색바 또는 이슈 목록 페이지의 검색 입력란에 검색어를 입력한다.
2. 시스템이 PostgreSQL FTS + GIN 인덱스를 활용하여 제목, 설명, 라벨에 대해 전문 검색을 수행한다.
3. 한국어 형태소 분석이 적용되어 검색 정확도가 향상된다.
4. 검색 결과가 관련도 순으로 표시되며, 검색어가 하이라이팅된다.
5. 사용자가 결과에서 원하는 이슈를 클릭하면 상세 페이지로 이동한다.

**대안 플로우:**

- **4a.** 검색 결과가 0건이면, "검색 결과가 없습니다" 메시지와 검색어 수정 제안을 표시한다.
- **1a.** 글로벌 검색바에서 검색 시, 이슈와 정책 결과가 탭으로 구분되어 표시된다.

**사후 조건:** 사용자가 검색 조건에 맞는 이슈를 확인한다.

**관련 화면:** `/issues`

#### UC-ISS-007: 이슈 워치

| 항목 | 내용 |
|------|------|
| **UC ID** | UC-ISS-007 |
| **이름** | 이슈 워치 (팔로우) |
| **액터** | Member, Manager, Admin, Viewer |
| **사전 조건** | 대상 이슈가 존재한다. |

**기본 플로우:**

1. 사용자가 이슈 상세 페이지에서 `[워치]` 버튼을 클릭한다.
2. 시스템이 해당 사용자를 이슈의 워치 목록에 추가한다.
3. 이후 해당 이슈에 변경(상태 변경, 댓글 추가, 담당자 변경 등)이 발생하면, 워치 사용자에게 알림이 발송된다.
4. `[워치]` 버튼이 `[워치 해제]`로 변경된다.

**대안 플로우:**

- **1a.** 이미 워치 중인 경우, `[워치 해제]`를 클릭하면 워치 목록에서 제거되고 알림이 중단된다.
- **3a.** 알림 채널은 사용자의 알림 설정(UC-NTF-001)에 따라 결정된다.

**사후 조건:** 사용자가 이슈의 변경 알림을 수신하거나 수신을 중단한다.

**관련 화면:** `/issues/{id}`

#### UC-ISS-008: 유사 이슈 추천 (생성 시 중복 탐지)

| 항목 | 내용 |
|------|------|
| **UC ID** | UC-ISS-008 |
| **이름** | 유사 이슈 추천 (생성 시 중복 탐지) |
| **액터** | Member, Manager, Admin |
| **사전 조건** | 사용자가 이슈 생성 폼에서 제목과 설명을 입력한 상태이다. |

**기본 플로우:**

1. 사용자가 이슈 생성 폼에서 제목을 입력하고 포커스를 이동하거나 `[생성]`을 클릭한다.
2. 시스템이 입력된 제목과 설명을 pgvector 임베딩으로 변환한다.
3. 기존 이슈의 임베딩과 코사인 유사도를 비교한다.
4. 유사도 임계값(예: 0.85) 이상인 이슈가 있으면, "유사한 이슈가 존재합니다" 패널이 표시된다.
5. 패널에 유사 이슈 목록(제목, 상태, 유사도 %)이 표시된다.
6. 사용자가 유사 이슈를 클릭하여 확인하거나, "새로 생성" 버튼으로 생성을 진행한다.

**대안 플로우:**

- **4a.** 유사 이슈가 없으면, 패널을 표시하지 않고 바로 생성을 진행한다.
- **6a.** 사용자가 유사 이슈에 "중복" 관계를 설정하면, 두 이슈가 `duplicates` 관계로 링크된다.

**사후 조건:** 중복 이슈 생성이 방지되거나, 명시적으로 중복임을 인지하고 생성한다.

**관련 화면:** `/issues/new`

#### UC-POL-001: 정책 작성

| 항목 | 내용 |
|------|------|
| **UC ID** | UC-POL-001 |
| **이름** | 정책 작성 (Markdown 에디터) |
| **액터** | Manager, Admin |
| **사전 조건** | 사용자에게 정책 관리 권한이 있다. |

**기본 플로우:**

1. 사용자가 좌측 메뉴 `[정책]` > `[새 정책 작성]` 버튼을 클릭한다.
2. 시스템이 정책 유형 선택 화면을 표시한다 (보안 정책, 운영 정책, 기타 또는 빈 양식).
3. 사용자가 유형을 선택하면, 해당 템플릿이 적용된 Markdown 에디터가 열린다.
4. 좌측에 Markdown 편집기, 우측에 실시간 미리보기가 표시된다.
5. 사용자가 정책 제목, 본문, 유효기간, 대상 부서/팀을 입력한다.
6. `[임시 저장]` 버튼으로 DRAFT 상태로 저장하거나, `[검토 요청]`으로 REVIEW 상태로 전환한다.
7. 시스템이 정책을 저장하고, REVIEW 전환 시 승인자에게 알림을 발송한다.

**대안 플로우:**

- **3a.** 빈 양식을 선택하면, 빈 Markdown 에디터가 열린다.
- **6a.** 필수 항목(제목, 본문)이 비어 있으면 저장을 차단하고 유효성 오류를 표시한다.

**사후 조건:** 정책 문서가 DRAFT 또는 REVIEW 상태로 저장된다.

**관련 화면:** `/policies/new`, `/policies/{id}/edit`

| 항목 | 내용 |

| **액터** | 전사 임직원 (모든 역할) |
| **사전 조건** | 게시된 정책이 1건 이상 존재하며, 임베딩이 생성되어 있다. |

- **4a.** LLM 호출에 실패하면, pgvector 검색 결과(관련 정책 링크 목록)만 폴백으로 반환한다.

**사후 조건:** 사용자가 정책 기반의 답변을 수신한다.

#### UC-CON-004: 동기화 충돌 해결

| 항목 | 내용 |
|------|------|
| **UC ID** | UC-CON-004 |

6. `[해결]` 버튼을 클릭한다.
7. 시스템이 선택된 값으로 이슈를 업데이트하고, 양쪽 플랫폼에 동기화한다.
8. `sync_status`가 `SYNCED`로 변경된다.

**대안 플로우:**

- **5a.** "외부 값 사용" 또는 "내부 값 사용" 일괄 적용 버튼으로 전체 필드를 한 번에 선택할 수 있다.

**사후 조건:** 충돌이 해결되고, 양쪽 플랫폼의 데이터가 일치한다.

2. 일반 설정: 조직명, 로고, 기본 언어, 시간대를 설정한다.
3. 보안 설정: 비밀번호 정책, 세션 타임아웃, 로그인 시도 제한을 설정한다.
4. 이슈 설정: 기본 이슈 상태 워크플로우, 커스텀 필드를 설정한다.
5. `[저장]` 버튼을 클릭한다.

**대안 플로우:**

- **3a.** 보안 설정 변경 시, 변경 전 현재 비밀번호 재확인을 요구한다.

**사후 조건:** 시스템 전역 설정이 업데이트된다.

**관련 화면:** `/settings`
