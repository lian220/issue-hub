# IssueHub - 사용자 시점 워크플로우 (User Flow)

> 문서 버전: 2.0
> 최종 수정일: 2026-03-21
> 작성자: IssueHub AI 어시스턴트 (전문가 리뷰 기반 도출)
> 상태: 초안 (Draft)

---

## 1. 개요 (Overview)

이 문서는 기존 PRD, 시스템 아키텍처 및 로드맵 문서에는 구체적으로 명시되어 있지 않았던 **"사용자(User) 시점에서의 화면 흐름과 여정(User Journey)"**을 구체화한 문서입니다. 시스템 간의 데이터 흐름(Data Flow)이 아닌, **실제 조직 구성원들이 IssueHub를 화면에서 어떻게 클릭하고 인지하며 상호작용하는지**를 7가지 핵심 페르소나 케이스로 나누어 설명합니다.

### 시나리오 요약

| Case | 페르소나 | 시나리오 | Phase |
|------|---------|---------|-------|
| 1 | 운영/CS 담당자 | 버그 접수 → 자동 라우팅 → 해결 알림 | Phase 1-2 |
| 2 | 보안 관리자 | 정책 작성 → 승인 → 게시 → 읽음 확인 → AI 위반 감지 | Phase 3 |
| 3 | PM/기획자 | 대시보드 → SLA 모니터링 → 워크로드 리밸런싱 | Phase 2 |
| 4 | 관리자 | 커넥터 설정 → 자동화 규칙 생성 → 테스트 | Phase 1-2 |
| 5 | 개발 리드 | GitHub PR → 이슈 연결 → 코드 리뷰 → 자동 상태 변경 | Phase 2 |
| 6 | 신규 입사자 | 계정 생성 → 팀 배정 → 정책 읽음 확인 → 첫 이슈 할당 | Phase 2-3 |
| 7 | 컴플라이언스 담당 | 정책 만료 감지 → 갱신 → 감사 보고서 → 위반 추적 | Phase 3 |

## 2. 핵심 유저 페르소나 기반 시나리오

### Case 1: 운영/CS 담당자의 버그 접수 및 자동 라우팅 흐름 (Slack ➔ IssueHub ➔ Jira)

> **페르소나**: 커뮤니티 매니저(CS) 및 결제 백엔드 개발 리드
> **목적**: 흩어진 채널에서 들어온 VOC를 빠르게 개발팀에 전달하고 진행 상황을 투명하게 추적한다.

1. **이슈 접수 (Slack/Discord)**:
   - CS 담당자가 커뮤니티 채널에서 "결제 진행 시 타임아웃 오류가 발생한다"는 유저 제보를 확인합니다.
   - 메신저 입력창에 `/issue create "결제 타임아웃 버그"`를 타이핑하거나, 메시지의 `[이슈로 만들기]` 인터랙티브 버튼을 클릭합니다.
2. **이슈 정제 및 자동화 개입 (IssueHub 엔진)**:
   - 메신저의 모달 창을 통해 상세 내용을 입력하고 `[제출]` 버튼을 누릅니다.
   - **(자동화 엔진 트리거)**: 내용을 분석한 IssueHub 엔진이 "결제" 키워드를 감지하여 우선순위를 `CRITICAL`로 변경하고, 대상 프로젝트를 결제 백엔드팀(Jira: PAY 프로젝트)으로 자동 지정합니다.
3. **개발 리드의 인지 및 배정 (대시보드)**:
   - 결제팀 개발 리드는 IssueHub의 **칸반 보드 뷰(Kanban Board View)**에서 `CRITICAL` 상태로 붉은색 알림이 깜빡이는 티켓을 발견합니다.
   - 드래그 앤 드롭으로 티켓을 특정 개발자(Assignee)에게 끌어다 놓습니다. 이 동작은 즉시 Jira 에 동기화(양방향 Sync)됩니다.
4. **이슈 해결 및 역방향 알림 (GitHub ➔ IssueHub ➔ Slack)**:
   - 담당 개발자가 코드를 수정하고 GitHub에 PR을 올리면서 `#PAY-102` (이슈 티켓 번호) 태그를 답니다.
   - 코드가 Merge되는 순간, GitHub 웹훅을 통해 IssueHub 티켓 상태가 `RESOLVED`로 자동 변경됩니다.
   - CS 담당자의 Slack 채널(또는 DM)에 즉시 "결제 타임아웃 버그가 해결되었습니다."라는 봇 알림 카드(Adaptive 카드의 형태)가 수신되어, 고객에게 빠르게 피드백을 전달합니다.

### Case 2: 보안 관리자의 정책 배포 및 컴플라이언스 준수 흐름

> **페르소나**: 보안/컴플라이언스 담당관 및 전사 임직원
> **목적**: 정책을 체계적으로 배포하고, 모든 구성원이 읽었는지 증적을 남기며, AI를 통해 정책 위반을 사전에 감지한다.

1. **정책 초안 생성 및 승인 (IssueHub 정책 탭)**:
   - 보안 담당관이 IssueHub 좌측 메뉴 `[정책 (Policies)]` 탭에 진입하여 `[새 정책 작성]` 버튼을 클릭합니다.
   - 마크다운 에디터에서 '26년도 상반기 개인정보 취급 방침 가이드'를 작성하고 상태를 `DRAFT`에서 `REVIEW`로 변경한 뒤, CISO(정보보호최고책임자)를 승인권자로 지정합니다.
   - CISO가 알림을 클릭해 내용을 확인 후 `[승인(APPROVE)]` 버튼을 누르면 최종적으로 `PUBLISHED(게시됨)` 상태가 됩니다.
2. **읽음 확인 프로세스 (Read-receipt)**:
   - 게재됨과 동시에 전사 직원의 Teams/Slack 채널에 정책 확인 요청 알림이 발송됩니다.
   - 직원은 IssueHub 정책 페이지 링크로 들어와 문서를 끝까지 스크롤한 뒤 `[읽음 확인]` 버튼을 명시적으로 클릭해야 합니다.
   - 보안 담당관은 대시보드의 **컴플라이언스 준수율 차트**에서 '개발팀 읽음 60% / 마케팅팀 90%' 식별하며 독려를 진행합니다.
3. **AI 기반 정책 위반 감지 (Phase 3 기능)**:
   - 이후 한 신입 개발자가 버그를 추적하기 위해 고객의 `주민등록번호`가 포함된 시스템 로그를 IssueHub 티켓 Description에 그대로 복사/붙여넣기 합니다.
   - 이슈 저장 버튼을 누르는 순간, 내부 AI(pgvector RAG)가 정책 위반 가능성을 감지하고 "⚠️ 입력하신 내용에 개인정보(주민등록번호)가 포함된 것으로 보입니다. 저장을 계속하시겠습니까?" 라는 **경고 팝업 모달**을 띄워 인적 오류(Human Error)를 방지합니다.

### Case 3: PM/기획자의 일간 프로젝트 진척도 모니터링 흐름

> **페르소나**: 프로젝트 매니저 (PM)
> **목적**: 분산된 모든 팀의 업무 진행 상황과 병목(Bottleneck) 구간을 한눈에 파악한다.

1. **모닝 브리핑 및 통합 대시보드 로드**:
   - PM이 데일리 스탠드업 미팅 전 IssueHub의 `[통합 대시보드]` 페이지에 접속합니다.
   - Jira(백엔드팀), GitHub(프론트엔드팀), 디자인 전용 툴에서 연동된 이슈가 **단일 타임라인 차트**로 노출됩니다.
2. **SLA 모니터링 및 병목 뷰어 확인**:
   - 위젯 중 **`SLA 위반(Breached)`** 항목에 집중합니다.
   - 고객의 문의가 들어오거나 우선순위가 높은 작업이 24시간 넘게 `OPEN` 상태에 머물러 있는 리스트(전체 빨간색으로 하이라이팅)를 클릭하여 필터링합니다.
3. **팀 워크로드 분산 및 리-어사인 (Re-assign)**:
   - SLA 위반 티켓 5개가 모두 특정 A 개발자에게 몰려 있는 것을 하단 **팀 워크로드 차트**를 통해 시각적으로 발견합니다.
   - PM은 해당 개발자에게 지정된 티켓 박스들을 마우스로 드래그(Drag)하여 유휴 상태인 B 개발자의 Lane으로 이동시킵니다.
   - 이 변경 사항은 백그라운드 환경에서 각 플랫폼(GitHub/Jira)의 Assignee 변경 API를 호출해 자동으로 동기화됩니다.

### Case 4: 관리자의 커넥터 설정 및 자동화 규칙 생성 흐름

> **페르소나**: 시스템 관리자 (Admin)
> **목적**: IssueHub와 외부 플랫폼을 연동하고, 이슈 자동 분류/라우팅 규칙을 설정한다.

1. **커넥터 설정 (Settings → Connectors)**:
   - 관리자가 좌측 메뉴 `[설정]` → `[커넥터 관리]`에 진입합니다.
   - `[+ 새 커넥터 추가]` 버튼을 클릭하면 플랫폼 선택 화면(Jira, GitHub, Slack, Teams, Discord)이 나타납니다.
   - Jira를 선택하면 OAuth 인증 플로우가 시작되어 Atlassian 로그인 → 권한 승인 → 콜백으로 돌아옵니다.
   - 인증 완료 후 동기화할 Jira 프로젝트를 체크박스로 선택하고 `[연동 시작]`을 클릭합니다.
   - 커넥터 상태 카드에 "연결됨 ✅ / 마지막 동기화: 방금 전"이 표시됩니다.

2. **필드 매핑 설정 (Connector → Field Mapping)**:
   - 연동된 커넥터 카드의 `[필드 매핑 설정]`을 클릭합니다.
   - 좌측에 Jira 필드, 우측에 IssueHub 필드가 나열된 매핑 화면이 표시됩니다.
   - 기본 매핑(summary→title, status→status 등)은 자동 설정되어 있고, 커스텀 필드는 드롭다운으로 매핑하거나 `metadata` JSONB에 보존하도록 선택합니다.
   - 상태 매핑 테이블에서 Jira의 "In Review" → IssueHub "IN_PROGRESS" 같은 커스텀 매핑을 설정합니다.

3. **자동화 규칙 생성 (Automation → Rules)**:
   - `[자동화]` 메뉴 → `[+ 새 규칙 만들기]`를 클릭합니다.
   - 3단계 빌더 화면이 나타납니다:
     - **트리거**: "이슈 생성 시" 선택
     - **조건**: "제목에 '결제' 포함 AND 소스가 Slack" 설정
     - **액션**: "우선순위를 CRITICAL로 변경 + 결제팀에 자동 배정 + Slack #payment-alerts 채널에 알림"
   - `[Dry-Run 테스트]` 버튼으로 최근 이슈 10건에 규칙을 시뮬레이션 → 결과 미리보기 확인
   - `[규칙 활성화]`로 저장합니다.

4. **규칙 모니터링**:
   - 자동화 대시보드에서 각 규칙의 실행 횟수, 성공/실패율, 최근 실행 이력을 확인합니다.
   - 실패한 실행 건을 클릭하면 상세 로그(트리거 시점, 조건 평가 결과, 액션 실행 결과)를 볼 수 있습니다.

### Case 5: 개발 리드의 GitHub PR-이슈 연결 및 코드 리뷰 흐름

> **페르소나**: 개발 리드 (Tech Lead)
> **목적**: GitHub의 PR과 IssueHub 이슈를 자동 연결하고, 코드 리뷰 상태를 이슈에 실시간 반영한다.

1. **이슈 확인 및 브랜치 생성**:
   - 개발 리드가 IssueHub 이슈 상세 페이지에서 `IH-234: API 응답 속도 개선` 이슈를 확인합니다.
   - 이슈 상세의 `[브랜치 이름 복사]` 버튼을 클릭하면 `feature/IH-234-api-response-optimization`이 클립보드에 복사됩니다.
   - 터미널에서 해당 이름으로 브랜치를 생성하고 작업합니다.

2. **PR 생성 및 자동 연결 (GitHub)**:
   - 개발자가 GitHub에서 PR을 생성할 때 본문에 `Fixes IH-234` 또는 `Closes #IH-234`를 포함합니다.
   - GitHub 웹훅이 발동되어 IssueHub가 PR 번호와 이슈를 자동 연결합니다.
   - IssueHub 이슈 상세 페이지의 **"연결된 PR"** 섹션에 PR 링크, 작성자, 상태가 실시간 표시됩니다.

3. **코드 리뷰 상태 연동**:
   - 리뷰어가 GitHub에서 "Changes Requested"를 남기면, IssueHub 이슈 상태가 자동으로 `IN_PROGRESS` 상태를 유지하며 "리뷰 변경 요청" 태그가 추가됩니다.
   - 리뷰어가 "Approved"를 남기면 "리뷰 승인됨" 태그로 업데이트됩니다.
   - 개발 리드의 칸반 보드에서 각 이슈 카드에 리뷰 상태 아이콘(🔴 변경요청 / 🟢 승인)이 표시됩니다.

4. **머지 및 자동 상태 변경**:
   - PR이 머지되는 순간 GitHub 웹훅 → IssueHub 이슈 상태가 `RESOLVED`로 자동 전환됩니다.
   - 이슈를 생성했던 CS 담당자 또는 보고자에게 Slack/Teams 알림이 발송됩니다.
   - CI/CD 파이프라인 연동 시(Phase 4), 배포 완료 후 `CLOSED`로 최종 전환됩니다.

### Case 6: 신규 입사자의 온보딩 흐름

> **페르소나**: 신규 입사 개발자 및 팀 리드/관리자
> **목적**: 신규 입사자가 시스템에 접근하고, 필수 정책을 확인하고, 첫 업무를 시작할 수 있도록 안내한다.

1. **계정 생성 및 팀 배정 (관리자)**:
   - 관리자가 `[설정]` → `[사용자 관리]` → `[+ 사용자 초대]`를 클릭합니다.
   - 신규 입사자의 이메일을 입력하고, 역할(`Member`), 소속 팀(`백엔드팀`)을 선택하여 초대합니다.
   - 신규 입사자에게 이메일로 초대 링크가 발송됩니다.

2. **첫 로그인 및 프로필 설정 (신규 입사자)**:
   - 초대 링크를 클릭하면 비밀번호 설정 → 프로필 입력(표시 이름, 프로필 사진) 화면이 나타납니다.
   - 외부 플랫폼 계정 연결 화면에서 Jira Account ID, GitHub Username을 입력합니다 (이후 동기화 시 담당자 자동 매핑에 사용).
   - 알림 설정에서 선호 채널(Slack DM / 이메일 / Teams)과 다이제스트 주기를 선택합니다.

3. **필수 정책 읽음 확인**:
   - 첫 로그인 시 **"읽지 않은 필수 정책"** 배너가 상단에 노출됩니다.
   - 배너를 클릭하면 미확인 정책 목록이 표시됩니다 (예: "개인정보 취급 방침", "코드 리뷰 가이드라인").
   - 각 정책을 열어 읽은 뒤 `[읽음 확인]` 버튼을 클릭해야 배너가 사라집니다.
   - 팀 리드와 컴플라이언스 담당관의 대시보드에서 신규 입사자의 읽음 진행률이 실시간 반영됩니다.

4. **첫 이슈 할당 수신**:
   - 팀 리드가 칸반 보드에서 온보딩용 이슈(`IH-300: 개발 환경 셋업`)를 신규 입사자에게 드래그합니다.
   - 신규 입사자에게 설정한 채널로 "새 이슈가 배정되었습니다" 알림이 도착합니다.
   - 이슈 상세 페이지에서 설명, 관련 문서 링크, 예상 마감일을 확인하고 작업을 시작합니다.

### Case 7: 컴플라이언스 담당관의 정책 갱신 및 감사 흐름

> **페르소나**: 컴플라이언스 담당관
> **목적**: 만료 예정 정책을 갱신하고, 조직 전체의 정책 준수 현황을 감사 보고서로 관리한다.

1. **만료 예정 정책 알림 수신**:
   - IssueHub가 정책 만료 30일/14일/7일 전에 단계별 알림을 발송합니다.
   - 컴플라이언스 담당관의 Slack DM에 "⚠️ '원격근무 보안 정책'이 14일 후 만료됩니다. 갱신이 필요합니다."라는 알림이 도착합니다.

2. **정책 갱신 (새 버전 생성)**:
   - `[정책]` 탭에서 만료 예정 정책을 클릭하고 `[새 버전 작성]` 버튼을 누릅니다.
   - 이전 버전 내용이 에디터에 자동 로드되어, 변경 사항만 수정합니다.
   - 변경 요약(Change Summary)을 작성하고 `REVIEW` 상태로 전환합니다.
   - 승인 워크플로우가 시작됩니다 (과반수 승인 → PUBLISHED).

3. **컴플라이언스 대시보드 확인**:
   - `[대시보드]` → `[컴플라이언스]` 탭에서 전체 정책 현황을 조회합니다.
   - 카드 형태로 표시되는 주요 지표:
     - 전체 정책 수 / 활성 / 만료 / 갱신 중
     - 정책별 읽음률 (막대 차트)
     - 팀별 준수율 히트맵
     - 최근 위반 감지 건수

4. **감사 보고서 생성**:
   - `[CSV/Excel 내보내기]` 버튼으로 특정 기간의 데이터를 다운로드합니다:
     - 정책별 버전 이력 + 승인자 + 승인일시
     - 구성원별 읽음 확인 일시
     - 위반 감지 이력 + 자동 생성된 이슈 목록
   - 규제 감사 시 이 보고서를 증적 자료로 제출합니다.

5. **위반 이슈 추적**:
   - 컴플라이언스 대시보드에서 "최근 위반 감지" 섹션의 이슈를 클릭합니다.
   - 자동 생성된 이슈 상세에서 위반 유형, 관련 정책, 담당자, 조치 상태를 확인합니다.
   - 미조치 건에 대해 에스컬레이션 규칙이 적용되어 리드 → 매니저 순으로 알림이 발송됩니다.
